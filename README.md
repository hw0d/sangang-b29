# LSSCPATF Records System

Los Santos STEP and Criminal Profiteering Act Task Force — a gang and
organized-crime records system: group profiles, individual case files with
a mugshot, a tattoo / marking gallery per person, and known-affiliate links
between people.

## Stack

- Next.js 16 (App Router) + TypeScript + Tailwind CSS
- PostgreSQL + Prisma
- NextAuth (credentials login) for the password-protected admin dashboard
- Uploaded images (mugshots, tattoo photos, group symbols) are stored as
  blobs in Postgres, not on disk — Render's web service filesystem is
  ephemeral and wiped on every deploy/restart, so disk storage would lose
  images constantly

## Local development

Prerequisites: Node 20+, a PostgreSQL database (local or remote).

```bash
npm install
cp .env.example .env
# edit .env: set DATABASE_URL to your Postgres instance, set AUTH_SECRET
# (generate one with `openssl rand -base64 32`), and set
# SEED_ADMIN_USERNAME / SEED_ADMIN_PASSWORD to your choice

npm run db:migrate   # applies prisma/migrations to your database
npm run db:seed      # creates the admin user + example data

npm run dev
```

Visit `http://localhost:3000` for the public site and `/admin/login` to sign
in with the admin credentials you set in `.env`.

## Deploying to Render

This repo includes a `render.yaml` Blueprint that provisions everything in
one go: a free PostgreSQL database and a free Node web service, wired
together automatically.

1. Push this repo to GitHub.
2. In the Render dashboard, choose **New > Blueprint** and point it at the
   repo. Render will read `render.yaml` and show you the resources it's
   about to create (`sangang-db` + `sangang-web`).
3. Render will prompt you for the two `sync: false` environment variables —
   `SEED_ADMIN_USERNAME` and `SEED_ADMIN_PASSWORD`. Set them to whatever
   you want the first admin login to be.
4. Click **Apply**. Render provisions the database, then builds and deploys
   the web service. The start command runs, in order: `prisma migrate
   deploy` (keeps the schema in sync — free-tier services don't support
   Render's `preDeployCommand`, so this runs at startup instead, and
   re-running it against an up-to-date schema is a safe no-op), then a
   script that upserts the admin account from the `SEED_ADMIN_USERNAME` /
   `SEED_ADMIN_PASSWORD` env vars you set in step 3, then `next start`.
5. Sign in at `https://<your-service>.onrender.com/admin/login` with the
   username/password from step 3.

Because that admin-account step runs automatically on every boot, it works
on the free plan even though **Render's free plan has no Shell/SSH
access** — there's nothing to run manually. If you ever change
`SEED_ADMIN_USERNAME`/`SEED_ADMIN_PASSWORD` in the dashboard, saving them
triggers a redeploy, which re-provisions the account with the new values on
the next boot.

Want the example groups/profiles too (not just the admin account)? That
still requires `npm run db:seed`, which needs either Shell access (paid
plans) or pointing your local `.env`'s `DATABASE_URL` at the database's
**External Database URL** (from the Render Postgres dashboard) and running
`npm run db:seed` from your own machine. Only run it once — unlike the
admin-account bootstrap, the example data isn't safe to insert twice.

Notes:

- Render's **free** Postgres plan is deleted after a period of inactivity /
  after its expiry window — check the current Render free-tier database
  policy before relying on it long-term, and upgrade the `plan:` in
  `render.yaml` (or in the dashboard) if you need persistence guarantees.
- The free web service instance spins down after inactivity and cold-starts
  on the next request — expect a slow first load after idle periods.
- `AUTH_SECRET` is auto-generated by Render via `generateValue: true`; you
  don't need to set it yourself.

## Data model

- **Group** — a gang/crew: name, aliases, description, territory, colors,
  status (active/disbanded), a symbol image, and its members.
- **Profile** — an individual case file: name, alias, status (at large / in
  custody / probation / deceased / unknown), physical description, a
  mugshot, optional group affiliation + rank, tattoos, and affiliate links.
- **Tattoo** — a photo + body location + meaning, attached to a profile.
- **ProfileLink** — a directional relationship between two profiles
  (associate / rival / family / subordinate-of / leader-of). Each link only
  needs to be created once from either side — the inverse relationship
  (e.g. "subordinate of" for "leader of") is computed automatically when
  rendering the other profile's page.

## Admin dashboard

`/admin` (behind login) provides CRUD for groups and profiles, including:

- Uploading a symbol image (groups) or mugshot (profiles)
- Adding/removing tattoos with their own photo, body location, and meaning
- Linking/unlinking known affiliates with a relationship type and note

Only create additional admin accounts by adding rows to `AdminUser`
(password hashed with bcrypt) — there's no public sign-up.

## Live updates

Every open tab (public or admin) keeps a Server-Sent Events connection open
to `/api/events`. Whenever an admin action creates, edits, or deletes a
group/profile/tattoo/affiliate link, the server broadcasts a "something
changed" signal and every connected tab silently re-fetches its current
page's data (`router.refresh()`) — no polling, no manual reload needed to
see other people's changes show up.

This uses a plain in-process `EventEmitter` (`src/lib/eventBus.ts`), which
is enough as long as the app runs as a single instance (true for Render's
free web-service plan). If this is ever scaled to multiple instances, the
event bus would need to move to something all instances can share (e.g.
Postgres `LISTEN`/`NOTIFY`), since in-memory events don't cross processes.
